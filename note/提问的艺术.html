提问的艺术

2022-11-21

前言

在真正开始一篇读后感之前，我突然好奇，我的第一次提问是什么样的。于是我在百度知道上找到了我提出的第一个问题：

较真的分析一下，也不是说这个提问本身不好，毕竟问题太过简单怎么问其实都不为过，二者也不能对小学生的我苛求太多，不过这里最大的问题是他不应该在互联网上被问出来，因为这完全可以通过搜索“赞美他人的成语”，或者翻阅自己的笔记来解决。

所以，提问的最佳实践其实是不提问，因为通用的问题大部分已经通过互联网得到解答，而过于具体专业的问题，我倾向于线下或者在专业论坛提问，这样得到回答的概率也会大很多。

当然有个前提是建立在互联网上，另外得是技术问题，千万不要带入现实生活中的聊天，比如别人打电话问你“你那边天气怎么样”，你回他一句“蠢货，天气好不好不会上网查？STFW去！”。那么我相信以后没人会给你打电话了。

正文

提问的智慧这篇文章太长了，很多都是对一些逆天行为作出指引（我是无法想象为什么要在提问时写个“紧急”）。所以，我针对一些自己的看法做了总结。

另外，我也对如何搜索问题和如何回答问题做了一些自己的看法。

搜索引擎

为什么不推荐用国内搜索引擎

  1. 国内搜索引擎不是精度不够，而是复制粘贴的回答太多了，CSDN罪大恶极。

  2. github issue、官方论坛等大部分使用英文。

  3. 国内缺乏较为统一的、面向大众技术网站。

自身技术水平

必须认识到技术水平会影响我们的问题，比如一个C语言新手提问“如何在linux环境下编译C语言代码”，我认为回答他STFM也是不负责任的行为。

作为提问者，应该思考问题的产生原因，是否出于自己知识水平限制，如果通过某方面的学习，是否可以解决或从新的角度看待我所遇到的问题。

作为回答者，也应该认识到提问者可能并非不会提问，他可能不知道linux是操作系统，不知道gcc工具，不知道C语言的编译过程，这些知识可以通过深入学习解决，但回答应该对初学者报以宽容。

时间与版本

因为技术迭代，很多技术问题可能经过1、2年的时间就已经不再使用，请小心甄别。

最后

提问前

  • 尝试在你准备提问的论坛的旧文章中搜索答案。
  • 尝试上网搜索以找到答案。
  • 尝试阅读手册以找到答案。
  • 尝试阅读常见问题文件（FAQ）以找到答案。
  • 尝试自己检查或试验以找到答案。
  • 向你身边的强者朋友打听以找到答案。
  • 如果你是程序开发者，请尝试阅读源代码以找到答案。

提问时

  • 仔细、清楚地描述你的问题或 Bug 的症状。
  • 描述问题发生的环境（机器配置、操作系统、应用程序、以及相关的信息），提供经销商的发行版和版本号（如：Fedora Core 4、Slackware 9.1等）。
  • 描述在提问前你是怎样去研究和理解这个问题的。
  • 描述在提问前为确定问题而采取的诊断步骤。
  • 描述最近做过什么可能相关的硬件或软件变更。
  • 尽可能地提供一个可以重现这个问题的可控环境的方法。
